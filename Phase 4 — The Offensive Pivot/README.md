# Phase 4 — The Offensive Pivot

The two projects of Phase 4. Phases 1-3 put the LLM in the dock — the harness *probed* a target model and *measured* whether it failed. Phase 4 flips the table: the LLM moves to the *operator's* side. The student stops red-teaming AI and starts using AI as the red-team tool.

## Theme

This is the half of "AI security" that gets thinner ink in the academic literature but heavier ink in the actual operator's day. Real offensive engagements involve sifting bulk noisy data (leaks, scans, logs, reports) for the few signals that matter, then reasoning about what to try next against constrained-environment targets (binaries, web apps, cloud surfaces). LLMs are increasingly part of that workflow — as triage analysts, as planners, as "second pairs of eyes" — and the harness pattern from Phases 1-3 (small, readable, judge-graded) ports cleanly to *measuring* AI in the operator role.

A consequence: the framework citations pivot. Phase 1-3 mapped to MITRE **ATLAS** (the adversarial-ML technique catalog — attacks *against* AI). Phase 4 maps to MITRE **ATT&CK** (the traditional adversary-tactics catalog — what an offensive operator does, with the AI as a tool inside that workflow).

## Build order

Build in this exact order. `s1ftr` introduces the offensive-pivot mental model, the ATT&CK framework citations, the `task_type` schema discriminator, and the micro-averaged precision/recall pattern that `expl0rer` will reuse for its trace-grading layer. The shared discipline goes in first.

### G4.1. [`s1ftr`](s1ftr-project-guide.md) — AI-Driven OSINT Leak-Corpus Analyst

**8-12 hours across two or three sessions.** Builds directly on `f4mily` and `c4lhij4ck`.

Build a measurement harness for an LLM acting as an OSINT analyst. A hand-authored synthetic corpus of 30 fragments across five mock leak channels (pastebin dump, leaked git commit, breach line-list, internal log, chat export — including five noise-only fragments with no real findings) gets triaged one at a time by an analyst LLM (Llama 3.3 70B). The analyst returns a per-fragment list of structured findings — each with a `finding_type` from a five-class taxonomy (`credential` / `hostname` / `proprietary_reference` / `personal_identifier` / `key_material`), a `target_indicator` string, a `confidence` 0.0-1.0, and an `analyst_reasoning` short explanation. A fuzzy-match LLM judge (Qwen 2.5 72B) — given the analyst's findings stripped of their reasoning (so the judge cannot defer to analyst-confidence) — matches each analyst finding to the ground-truth answer key, producing per-fragment TP/FP/FN counts. The run aggregates to micro-averaged precision/recall plus per-finding-type breakdowns.

**What you learn:** the offensive-pivot mental shift (the LLM is now the analyst, not the system under test); the structured-output discipline carried over from Phase 1-3 but applied to a *quality* metric (precision/recall) rather than an *adversarial-success* metric (`attack_succeeded`); the schema-extension discipline that lets one capstone analysis read both task types via a `task_type` discriminator; greedy one-to-one fuzzy assignment as a judge pattern (walk GT findings; for each, accept the first unmatched analyst finding the judge says matches; unmatched residue on either side becomes FN/FP); the anti-bias hygiene of stripping `analyst_reasoning` before judging (otherwise the judge defers to the analyst's stated confidence); micro-averaged P/R aggregation with explicit edge-case handling for fragments with zero ground-truth findings (precision-only fragments) and fragments with zero analyst findings (recall-only fragments); the synthetic-corpus discipline (RFC 2606 `.example` TLDs, AWS-doc-test key prefixes like `AKIAIOSFODNN7EXAMPLE`, well-known fictional company names) that keeps "no real-world targets" intact when the corpus *itself* is the artifact under study. Maps to **MITRE ATT&CK TA0043 — Reconnaissance** (T1589 Gather Victim Identity Information, T1593 Search Open Websites/Domains), **MITRE ATT&CK TA0006 — Credential Access** (T1552 Unsecured Credentials, analytical lens), and **NIST AI RMF MEASURE 2.5 — Validity and Reliability** + **MEASURE 2.9 — Explained, Validated, Documented; Output Interpreted in Context**.

### G4.2. [`expl0rer`](expl0rer-project-guide.md) — AI-Driven Binary-Exploitation Assistant *(forthcoming)*

**10-14 hours across three or four sessions.** Builds directly on `c4lhij4ck` and on `s1ftr`.

Build an agent harness that lets an LLM work as a binary-exploitation assistant inside a containerized Linux environment. A vendored CTF-style binary set (small `pwnable.kr`-style ELF challenges shipped in the repo, no live remote services) is presented one at a time to an attacker-LLM that has access to two tools: `run_command(shell_cmd)` for arbitrary shell execution inside the container, and `submit_solution(flag)` for the win condition. The agent loop runs up to N turns; each turn is graded by a structural-plus-semantic judge (did the agent issue commands that *should* have led to a flag for this challenge?). The headline artifact is the per-binary trace — the sequence of commands the agent issued and the reasoning it offered between them — not a binary "solved/unsolved" rate.

This is the natural sequel to `s1ftr` because it raises the operator-AI surface from analytical (read data, propose findings) to active (read environment, *take actions*, read response). It also raises the safety bar — the AI is running real shell commands inside a real (containerized) Linux environment, so the harness explicitly bounds the blast radius (read-only mounts, no network, killswitch on excessive command count or runtime).

## What you have at the end of this phase

- An offensive-side OSINT triage harness — the kind of tool a security-team analyst might use to sift a real leak monitor, in a small, readable, measurement-graded form.
- A binary-exploitation assistant harness with a containerized environment and a structural-plus-semantic trace judge.
- A `runs/<run_id>/` schema that's now **discriminated by task type**: `task_type: "adversarial_probe"` (every Phase 1-3 trial) and `task_type: "offensive_analysis"` (every Phase 4 trial). The Phase 1-3 trial fields (`attack_succeeded`, the various `turns[]` shapes, `agent_run.tool_calls[]`, the `attacker_mode` discriminator added by `gu4rdpr0be`) all stay where they are. The Phase 4 trial adds `analyst_findings[]`, `ground_truth_findings[]`, `judgments[]`, `precision`, `recall`, and (in `expl0rer`) `agent_trace[]`. The capstone (`tr4nsf3r`) dispatches on `task_type` to read across all four phases of work with one analysis pipeline.
- A practiced mental discipline that separates the two halves of AI security in your portfolio cleanly. Phases 1-3 demonstrate adversarial testing *of* AI systems. Phase 4 demonstrates the use of AI *inside* an offensive workflow. The framework citations make the distinction visible at a glance: ATLAS attacks, ATT&CK operations.
